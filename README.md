# Document Q&A Bot (RAG-powered)

A Retrieval-Augmented Generation system that answers questions strictly
from your own PDF/DOCX/TXT documents, with source-and-page citations, and
refuses to answer when the documents don't contain the answer.

🔗 **Live demo:** [Document Q&A Bot](https://documentappchatbot-peidh37uk5ujxpvgdefzzz.streamlit.app/)

## Architecture

```
                                 +--------------------+
                                 |  Custom Documents  |
                                 | (PDF, DOCX, TXT)   |
                                 +---------+----------+
                                           |
                                           v
+-------------------+            +---------+----------+            +------------------+
|    User Query      +---------->|   Vector Database   +---------->| Retrieved Chunks |
+---------+----------+            |  (Semantic Search)  |            +--------+---------+
          |                       +---------------------+                     |
          |                                                                   v
          |                       +--------------------+            +--------+---------+
          +---------------------->| Prompt Formulator  |<-----------+  Source Citations |
                                  +---------+----------+            +-------------------+
                                            |
                                            v
                                  +---------+----------+
                                  |  Language Model    |
                                  |   (Gemini)         |
                                  +---------+----------+
                                            |
                                            v
                                  +---------+----------+
                                  | Grounded Answer &  |
                                  |    Citations       |
                                  +--------------------+
```

## A note on the tech stack

The original assignment brief specifies `google-generativeai` and
`text-embedding-004`. Both have been retired by Google:

- `google-generativeai` is Google's own deprecated SDK — they now point
  everyone to the unified `google-genai` package.
- `text-embedding-004` was shut down by Google on **January 14, 2026**
  and now returns a `404 NOT_FOUND` error.

This project uses the current, supported replacements instead, so it
actually runs:

| Brief specifies | This project uses |
|---|---|
| `google-generativeai` | `google-genai` |
| `text-embedding-004` | `gemini-embedding-001` |

The RAG architecture, project structure, and all five pipeline steps from
the brief are otherwise implemented exactly as specified.

## Setup

1. **Install dependencies**

   ```bash
   pip install -r requirements.txt
   ```

2. **Configure your API key**

   Edit `.env` at the project root:

   ```
   GEMINI_API_KEY="your_actual_gemini_api_key_here"
   ```

3. **Add your own documents (optional)**

   Drop additional `.pdf`, `.docx`, or `.txt` files into `data/`. Sample
   documents (a business report, a science paper, and a product factsheet)
   are included so the project runs out of the box.

4. **Run ingestion** (only needed once, or whenever `data/` changes)

   ```bash
   python -m src.ingest
   ```

   This extracts text, chunks it, embeds the chunks with Gemini, and
   persists everything to `db/`. No re-embedding happens on later runs —
   `query.py` and `main.py` just load this database from disk.

5. **Ask questions**

   ```bash
   python -m src.main
   ```

   Or, for the optional bonus web UI:

   ```bash
   streamlit run app.py
   ```

## How It Works

### Step 2 — Document Ingestion & Text Extraction (`src/ingest.py`)
- `.pdf` files are parsed page-by-page with `pypdf`.
- `.docx` files have no native "page" concept, so paragraphs are grouped
  into pseudo-pages (~12 paragraphs each) for citation purposes.
- `.txt` files are treated as a single page.
- Every extracted unit becomes `{"text": ..., "source": filename, "page": N}`.

### Step 3 — Text Chunking Strategy
A small, dependency-free recursive character splitter tries natural
separators in order — paragraph breaks, then line breaks, then spaces,
then raw characters — so meaningful units stay intact wherever possible.
Default settings: `CHUNK_SIZE=1000` characters, `CHUNK_OVERLAP=150`
characters (configurable in `src/config.py`).

### Step 4 — Persisting the Vector Database
Chunks are embedded with `gemini-embedding-001` and written to a
**persistent local ChromaDB** collection at `./db`. Ingestion
(`ingest.py`) and querying (`query.py`/`main.py`) are deliberately
separate scripts — you only pay embedding costs once, at ingest time.

### Step 5 — Similarity Search & Retrieval (`src/query.py`)
The user's question is embedded with the same model/config used for the
chunks, then ChromaDB returns the top-k (`TOP_K=4` by default) closest
chunks by cosine similarity. Chunks below `MIN_SIMILARITY_THRESHOLD`
(0.3 by default) are dropped as irrelevant noise.

### Step 6 — Prompt Engineering & Answer Generation
Retrieved chunks are formatted with explicit `[Source: file, Page: N]`
labels and handed to Gemini under a strict system prompt that:
- Forbids answering from the model's own general knowledge.
- Requires an explicit "I cannot find the answer in the provided
  documents" response when the context doesn't cover the question.
- Requires inline citations in the format `(filename, Page X)`.

## Re-indexing

To force a full rebuild of the vector database (e.g. after editing
`CHUNK_SIZE` or adding new documents), just re-run:

```bash
python -m src.ingest
```

It clears and recreates the collection each time, so stale chunks never
linger.

## Adjusting Behavior

All tunable values live in `src/config.py`:
- `CHUNK_SIZE`, `CHUNK_OVERLAP` — chunking granularity.
- `TOP_K` — how many chunks are retrieved per question.
- `MIN_SIMILARITY_THRESHOLD` — relevance cutoff for retrieved chunks.
- `GENERATION_MODEL`, `EMBEDDING_MODEL` — swap Gemini models here.
